Automated Orchestration of Local-First Deterministic Development Environments: A Day-Zero Bootstrapper for High-Performance Windows Workstations
The emergence of Local-First Deterministic Development Environments (LDDE) represents a strategic shift in software engineering, moving away from fragmented, cloud-dependent toolchains towards integrated, hardware-bound ecosystems. For practitioners utilizing high-performance hardware such as the MSI Prestige 16 Studio, the objective is to eliminate the 'it works on my machine' paradox by creating a perfectly reproducible software stack directly on the host operating system. This report details the architectural requirements and the automated provisioning sequence necessary to transform a factory-reset Windows 11 machine into an LDDE, utilizing the Windows Subsystem for Linux (WSL2), containerised sandboxes, and local large language models (LLMs).
Architectural Foundations of the MSI Prestige 16 Studio LDDE
The MSI Prestige 16 Studio, powered by the Intel Core Ultra series, is uniquely positioned to handle the computational demands of an LDDE. Its architecture features dedicated Neural Processing Units (NPUs) and high-bandwidth memory (LPDDR5/x), which are essential for running local inference engines like Ollama without overwhelming the primary CPU cores1. A deterministic deployment ensures that the hardware's potential is not squandered by inefficient background processes or misconfigured environment paths.
In a 'Day-Zero' scenario, the system starts with no developer tools, requiring a bootstrapper that handles every stage of the lifecycle—from hypervisor initialisation to the final orchestration of agentic coding interfaces like OpenHands3. 	Hypervisor Initialisation and the Subsystem Layer
The core of any LDDE on Windows is the Virtual Machine Platform (VMP). Unlike traditional virtualisation, WSL2 uses a lightweight utility VM that provides a genuine Linux kernel while sharing resources dynamically with the host5. On a pristine MSI machine, these features are typically disabled. Enabling them via PowerShell requires interaction with the Deployment Image Servicing and Management (DISM) tool or the Enable-WindowsOptionalFeature cmdlet14.
While some components of WSL can be staged silently, a hard system reboot is technically unavoidable to initialise the Hyper-V hypervisor and the Virtual Machine Platform at the kernel level6. The bootstrapper is designed to detect these states and prompt the user for the necessary restart before proceeding with the user-mode toolchain.
Automated WSL2 Provisioning
The wsl --install command serves as a unified entry point, handling the feature enablement, kernel download, and distribution registration in a single pass5. For a deterministic environment, the script uses the --no-launch flag to prevent the distribution from opening a new window during the automated sequence, which would otherwise pause the script until the user manually exits the Linux shell16.

	Headless Dependency Management via Winget
The transition from a pristine OS to a developer-ready environment is facilitated by the Windows Package Manager (winget). On a factory-reset machine, winget provides a verified, single-source mechanism for tool acquisition, replacing the need for manual downloads1. The LDDE blueprint identifies four critical host-side dependencies: Docker Desktop for container orchestration, Ollama for local LLM inference, Git for source control, and Python 3.12 for local script execution and agent runtimes8.
Silent installation is paramount. Using the --silent and --accept-package-agreements flags allows the script to install complex software like Docker Desktop—which traditionally requires multiple GUI prompts—without user intervention1. This is essential for achieving the 'Day-Zero' objective of total automation.
External Storage Subsystem and Hardware Binding
A common failure point in local development is the saturation of the system C: drive by multi-gigabyte model weights and transient Docker volumes. The LDDE architecture resolves this by binding the environment to an external Nextech SSD13. This requires the bootstrapper to dynamically identify the drive using WMI (Windows Management Instrumentation) queries, looking for hardware-level identifiers such as manufacturer strings or model names12.
Once identified, the script must establish a persistent link. This involves creating a standard directory structure (OllamaModels for weights and OpenHandsWorkspace for active code) and configuring the OLLAMA_MODELS environment variable at the Machine scope13. This ensures that the Ollama service, which runs with system-level permissions, consistently references the external drive even across reboots or different user logins11.

	Local Inference and LLM Governance
The heart of the LDDE's intelligence layer is Ollama. By running the Qwen 2.5 Coder 7B model locally, the system achieves state-of-the-art coding assistance with zero latency and complete privacy2. The 7B parameter count, specifically the q4_K_M (4-bit quantization), is the 'Goldilocks' configuration for the MSI Prestige 16 Studio: it is large enough to retain complex reasoning for code repair and generation, yet small enough to fit within the 16GB–32GB RAM envelope typical of the device2.
The bootstrapper performs a headless ollama pull, which fetches the model layers directly from the Ollama library25. Because the environment variables were set in the previous phase, the multi-gigabyte download is automatically routed to the Nextech SSD, sparing the internal NVMe from wear and capacity issues13.
Orchestration of the OpenHands Agentic Workspace
OpenHands (formerly OpenDevin) acts as the bridge between the LLM and the local filesystem. It requires a sophisticated Docker configuration to function as a deterministic agent3. The docker-compose.yml file must be written directly to the external SSD to ensure the entire workspace is portable4.
Critical to this setup is the communication between the Docker container (running OpenHands) and the host-resident Ollama service. The configuration utilizes host.docker.internal to allow the containerized agent to query the host's GPU-accelerated inference engine27. Furthermore, the host's Docker socket (/var/run/docker.sock) is mounted into the OpenHands container, enabling it to spawn its own sandboxed runtimes for safe code execution4.
LDDE Master Bootstrapper: The Implementation Protocol
The following sections provide the pure, annotated PowerShell blocks required to achieve the LDDE state on a pristine MSI Prestige 16 Studio.
Block 1: Hypervisor and Subsystem Initialisation






PowerShell
# -----------------------------------------------------------------------------
# PHASE 1: HYPERVISOR & SUBSYSTEM INITIALIZATION
# -----------------------------------------------------------------------------

# Enable Windows Subsystem for Linux (WSL) and Virtual Machine Platform
# Note: dism.exe is utilized for maximum compatibility with pristine OS states [cite: 6, 15].
dism.exe /online /enable-feature /featurename:Microsoft-Windows-Subsystem-Linux /all /norestart
dism.exe /online /enable-feature /featurename:VirtualMachinePlatform /all /norestart

# <USER_ACTION_REQUIRED>
# A system reboot is technically unavoidable to initialize the hypervisor kernel [cite: 6, 14].
# Please REBOOT your MSI machine now and rerun this script to continue.
# -----------------------------------------------------------------------------

# Silent installation of the default Ubuntu distribution.
# --no-launch prevents the interactive setup from interrupting the script flow.
wsl --install -d Ubuntu --no-launch

# Ensure WSL 2 is the default version for performance consistency.
wsl --set-default-version 2

Block 2: Headless Dependency Installation (Zero UI)






PowerShell
# -----------------------------------------------------------------------------
# PHASE 2: HEADLESS DEPENDENCY INSTALLATION (WINGET)
# -----------------------------------------------------------------------------

# Verify winget is available and update the source metadata [cite: 9].
winget source update

# Array of target LDDE toolchain IDs for silent deployment.
$tools = @(
   "Docker.DockerDesktop",
   "Ollama.Ollama",
   "Git.Git",
   "Python.Python.3.12"
)

# Execute silent, headless installations with automated agreement acceptance.
foreach ($tool in $tools) {
   winget install --id $tool --silent --accept-package-agreements --accept-source-agreements
}

Block 3: External SSD Environment Binding






PowerShell
# -----------------------------------------------------------------------------
# PHASE 3: EXTERNAL SSD ENVIRONMENT BINDING
# -----------------------------------------------------------------------------

# Identify the Nextech SSD by querying the physical disk model string [cite: 12, 20, 23].
$nextechDisk = Get-WmiObject Win32_DiskDrive | Where-Object { $_.Model -match "Nextech" }

if ($null -eq $nextechDisk) {
   Write-Error "Nextech SSD not detected. Ensure drive is connected."
   return
}

# Traverse from Physical Disk to Logical Drive Letter [cite: 20].
$partition = Get-WmiObject -Query "ASSOCIATORS OF {Win32_DiskDrive.DeviceID='$($nextechDisk.DeviceID)'} WHERE AssocClass = Win32_DiskDriveToDiskPartition"
$logicalDisk = Get-WmiObject -Query "ASSOCIATORS OF {Win32_DiskPartition.DeviceID='$($partition.DeviceID)'} WHERE AssocClass = Win32_LogicalDiskToPartition"
$driveLetter = $logicalDisk.DeviceID # Dynamic detection, usually D: or E:

# Define LDDE directory structure on the external SSD.
$ollamaDataPath = Join-Path $driveLetter "OllamaModels"
$workspacePath = Join-Path $driveLetter "OpenHandsWorkspace"

# Create directories if they do not exist
if (-not (Test-Path $ollamaDataPath)) { New-Item -Path $ollamaDataPath -ItemType Directory -Force }
if (-not (Test-Path $workspacePath)) { New-Item -Path $workspacePath -ItemType Directory -Force }

# Set the OLLAMA_MODELS environment variable globally (Machine scope).
# This redirects all future model downloads to the Nextech SSD.
[System.Environment]::SetEnvironmentVariable("OLLAMA_MODELS", $ollamaDataPath, "Machine")
$env:OLLAMA_MODELS = $ollamaDataPath # Immediate session update

Block 4: Model Acquisition and Docker Initialization






PowerShell
# -----------------------------------------------------------------------------
# PHASE 4: MODEL ACQUISITION & DOCKER INITIALIZATION
# -----------------------------------------------------------------------------

# Trigger headless pull of the optimal high-precision coding model.
# This operation will store the ~4.7GB model on the external SSD.
ollama pull qwen2.5-coder:7b-instruct-q4_K_M

# Generate the LDDE blueprint docker-compose.yml on the external SSD.
$composeContent = @"
services:
 openhands:
   image: ghcr.io/all-hands-ai/openhands:latest
   container_name: openhands-app
   ports:
     - "3000:3000"
   environment:
     - SANDBOX_RUNTIME_CONTAINER_IMAGE=ghcr.io/all-hands-ai/runtime:0.50-nikolaik
     - WORKSPACE_MOUNT_PATH=$workspacePath
     - LLM_MODEL=ollama/qwen2.5-coder:7b-instruct-q4_K_M
     - LLM_BASE_URL=http://host.docker.internal:11434
   volumes:
     - /var/run/docker.sock:/var/run/docker.sock
     - $workspacePath:/opt/workspace_base
   extra_hosts:
     - "host.docker.internal:host-gateway"
   stdin_open: true
   tty: true
   restart: unless-stopped
"@

$composeFilePath = Join-Path $workspacePath "docker-compose.yml"
Set-Content -Path $composeFilePath -Value $composeContent -Force

# Execute docker compose to launch the OpenHands Task GUI in detached mode [cite: 31, 32].
Set-Location -Path $workspacePath
docker compose up -d

Architectural Analysis: Why this Protocol Succeeds
The success of the 'Day-Zero' bootstrapper lies in its strict adherence to the sequence of hardware-software associations. By enabling the Virtual Machine Platform first, the system is ready to receive the Linux kernel before any user-space applications are installed. The use of winget ensures that dependencies are not only installed but tracked by the Windows system for future updates1.
The dynamic detection of the Nextech SSD is the most critical logic gate. Without it, the environment would fail to be deterministic, as drive letters often shift when multiple peripherals are connected. By querying the Win32_DiskDrive class, the script anchors the LDDE to physical hardware rather than a potentially transient mount point12. This creates a 'Portable LDDE'—a developer can take the Nextech SSD to any machine running this bootstrapper and resume work instantly.
Token Dynamics and Inference Performance

A 4-bit quantized 7B model occupies approximately 4.7GB of space2. On the Prestige 16 Studio, with LPDDR5x memory providing upwards of 100 GB/s bandwidth, the model can achieve inference speeds that exceed standard reading speeds, making it ideal for real-time agentic interactions in OpenHands2.
Security and Isolation Protocols
In a deterministic development environment, security is achieved through isolation. OpenHands runs within a Docker container, ensuring that any code generated by the LLM is executed in a sandboxed environment (/opt/workspace_base) rather than on the host MSI machine4. The host's Docker socket is only shared with the OpenHands application itself, allowing it to manage the lifecycle of the sandbox without exposing the host OS to external threats4.
Furthermore, the LDDE maintains data sovereignty. By using Ollama for inference, no code snippets or prompts leave the MSI Prestige 16 Studio35. This is a critical requirement for enterprise-grade development where intellectual property protection is paramount.
Conclusion and Future Outlook
The automated provisioning of an LDDE on the MSI Prestige 16 Studio represents the ultimate integration of high-performance hardware and modern software orchestration. Through the strategic use of PowerShell, WSL2, and containerisation, the 'Day-Zero' bootstrapper transforms a factory-reset machine into a sophisticated engineering workstation in a matter of minutes.
The offloading of data to external Nextech SSD storage ensures that the system remains 'pristine' and high-performing, avoiding the degradation typical of long-term development setups. As local LLMs continue to evolve, particularly with the 32B and 72B variants of the Qwen 2.5 Coder series, this architectural foundation is ready to scale. The deterministic nature of the bootstrapper ensures that as hardware is upgraded, the development environment remains a constant, providing a stable platform for the next generation of software engineering.
